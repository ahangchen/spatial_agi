## 摘要

空中视觉语言导航（Aerial Vision-Language Navigation, AVLN）使无人机能够在复杂的3D环境中遵循自然语言指令进行导航。然而，现有的零样本AVLN方法面临多个挑战：单流视觉语言模型（VLM）决策不稳定、长程进度监控不可靠、以及安全性与效率之间的权衡问题。

本文提出**OnFly**，一个完全机载、实时的零样本AVLN框架。OnFly的核心创新包括：

1. **共享感知双智能体架构**：将高频目标生成与低频进度监控解耦，通过共享ViT主干网络和缓存视觉特征来消除冗余计算
2. **混合关键帧-最近帧记忆**：保留全局轨迹上下文的同时维持KV-cache前缀稳定性，实现可靠的长程监控
3. **语义-几何验证器**：使用VLM特征和深度信息验证VLM预测的目标，确保指令一致性和几何安全性
4. **滚动时域规划器**：在几何安全约束下生成优化的无碰撞轨迹

在仿真实验中，OnFly将任务成功率从最强基线的26.4%提升到**67.8%**，同时完全机载的真实世界飞行验证了其实时部署的可行性。

## 核心算法流程

### 系统架构概述

OnFly系统包含以下核心模块：

```
输入：自然语言指令 L
     ↓
[任务管理器] → 分解为子任务队列 {L₀, L₁, ..., Lₙ}
     ↓
[共享感知双智能体]
     ├─ [决策智能体] → 高频目标生成 → 候选目标 Gc
     └─ [监控智能体] → 低频进度监控 → 状态信号 sₜ
     ↓
[语义-几何验证器] → 目标验证与修正
     ↓
[滚动时域规划器] → 生成无碰撞轨迹
     ↓
输出：可执行飞行轨迹
```

### 1. 任务管理（Task Management）

#### 1.1 指令分解

给定长程指令 L，任务管理器使用机载VLM将其一次性分解为子任务序列：

```
L → {L₀, L₁, ..., Lₙ}
```

例如：
- 输入："飞到建筑物，找到二楼窗户，在那里停下"
- 分解：
  - L₀: "飞到建筑物"
  - L₁: "找到二楼窗户"
  - L₂: "在那里停下"

#### 1.2 子任务切换

当监控智能体输出稳定的**STOP**信号时，任务管理器推进到下一个子任务 Lᵢ₊₁，直到所有子任务完成。

---

### 2. 共享感知双智能体架构（Dual-Agent System）

#### 2.1 设计动机

传统单流VLM决策范式存在以下问题：
1. **频率需求耦合**：高频实时飞行控制与低频任务进度监控在同一推理流中互相干扰
2. **多目标错位**：两个任务共享相同上下文但追求不同目标，导致资源竞争
3. **决策不稳定**：低频分析阻塞高频目标生成，降低整体性能

#### 2.2 架构设计

```
                    共享ViT主干
                         ↓
                    缓存视觉特征
                    ↙         ↘
    [决策智能体]           [监控智能体]
    高频(实时)             低频(2秒/次)
    独立KV-cache          独立KV-cache
         ↓                      ↓
    目标点(px,py)          状态信号sₜ
         ↓                      ↓
    候选目标Gc            CONTINUE/STOP/LOST
```

**关键特性**：
- **共享感知**：两个智能体共享ViT主干网络，避免重复计算
- **独立推理**：各自维护独立的KV-cache，匹配不同的更新频率
- **异步执行**：监控智能体异步运行，不阻塞决策智能体

---

### 3. 决策智能体（Decision Agent）

#### 3.1 目标预测

决策智能体将导航决策建模为**图像空间目标点预测**：

```
(px, py) = VLM(P_dec, Lᵢ, Oₜ)
```

其中：
- P_dec：决策提示词
- Lᵢ：当前子任务
- Oₜ：当前观测 {rgbₜ, depthₜ, poseₜ}
- (px, py)：图像坐标下的目标点

#### 3.2 决策提示词设计

结构化的决策提示词包含：
1. **智能体角色定义**：明确决策智能体的职责
2. **任务上下文**：当前子任务描述
3. **输出格式约束**：严格要求返回整数(x,y)坐标
4. **执行约束**：基本飞行限制（如避障）
5. **历史点线索**：前一个3D目标在当前图像中的2D投影坐标（轻量级历史信息）

#### 3.3 候选目标封装

预测的目标点被封装为候选目标：

```
Gc = [px, py, depthₜ, fₜ, poseₜ]
```

其中：
- px, py：图像坐标
- depthₜ：当前深度图
- fₜ：缓存的ViT特征（用于后续语义验证）
- poseₜ：当前位姿（里程计）

---

### 4. 监控智能体与混合记忆（Monitoring Agent with Hybrid Memory）

#### 4.1 状态分类

监控智能体输出三种状态信号：

```
sₜ ∈ {CONTINUE, STOP, LOST}
```

- **CONTINUE**：当前子任务正常执行中
- **STOP**：当前子任务完成
  - 如果是最后一个子任务 → 终止整个episode
  - 否则 → 推进到下一个子任务
- **LOST**：偏离轨迹或丢失目标
  - 无人机先停止
  - 然后重新定向到上一个正常点的航向进行恢复

#### 4.2 监控提示词设计

监控提示词 P_mon 包含：
1. 智能体角色定义
2. 子任务上下文
3. 完成和失败标准（如目标到达/目标丢失）
4. 强制选择的输出格式

#### 4.3 混合记忆（Hybrid Memory）

**设计动机**：

传统滑动窗口记忆存在问题：
1. **全局上下文丢失**：新观测不断覆盖旧观测，长程轨迹信息逐渐丢失
2. **KV-cache前缀不稳定**：频繁覆盖早期记忆槽导致前缀变化，降低缓存命中率，增加推理延迟
3. **监控不可靠**：长程线索被覆盖，导致任务失败

**混合记忆结构**：

```
Mₜ = [o₁, Qₜ, oₜ]

其中：
- o₁：初始帧（固定）
- Qₜ：关键帧列表（长度S=4）
- oₜ：最新帧（实时更新）
```

#### 4.4 关键帧维护流程

**步骤1：关键帧候选生成**

```
每一步t：
1. 根据累积平移/旋转阈值生成候选集 Cₜ
2. 去重：Cₜ^new = Dedup(Cₜ, ε)
   - 相似度：ViT特征的余弦距离
3. 合并到池：Uₜ = Merge(Uₜ₋₁, Cₜ^new, ε)
```

**步骤2：关键帧选择**

```
1. 将遍历距离[0, Dₜ]划分为S个等长段
2. 将池中每个关键帧分配到对应段
3. 每段选择一个胜出者：
   - 如果前一个胜出者仍在同一段 → 重用（粘性规则）
   - 否则 → 选择最接近段中心的候选
4. 如果段为空 → 在序列化时填充
```

**步骤3：前缀稳定序列化**

```
构建有序关键帧列表 Qₜ：
1. 保留 Qₜ₋₁ 的最长有效前缀
2. 追加当前胜出者（去重）
3. 用未使用的近期池项填充剩余槽位
4. 按时间排序：Mₜ = [o₁, Qₜ, oₜ]
```

**优势**：
- ✅ 保留全局轨迹上下文（初始帧+关键帧）
- ✅ KV-cache前缀稳定（关键帧去重+有序序列化）
- ✅ 缓存重用率高，推理延迟低
- ✅ 可靠的长程监控（不会丢失早期信息）

### 5. 语义-几何验证局部规划（Semantic-Geometric Verified Local Planning）

#### 5.1 设计动机

VLM预测的目标点主要传达**语义意图**，不保证**几何可行性**：
- 可能落在障碍物上
- 可能在不可通行区域
- 纯几何修正可能违反指令语义

#### 5.2 验证流程

```
输入：候选目标 Gc = [px, py, depthₜ, fₜ, poseₜ]
     ↓
[步骤1] 语义引导目标修正
     ↓
[步骤2] 航向感知几何门控
     ↓
输出：验证后的3D导航目标 Gp
```

#### 5.3 语义引导目标修正

**算法流程**：

```python
def semantic_guided_refinement(px, py, depthₜ, fₜ):
    # 计算深度
    d = depthₜ[py, px]
    
    # 如果深度过大，跳过局部修正
    if d > D_max:
        return (px, py)
    
    # 计算相似度掩码（ROI内）
    M_p = {q ∈ ROI(p) | sim(fₜ[q], fₜ[p]) ≥ τ}
    
    # 连通区域分析（保留包含原始目标点的区域）
    C_p = connected_components(M_p, seed=(px, py))
    
    # 深度可行性检查
    Sₜ = threshold(depthₜ) + dilation(r)
    # Sₜ是障碍物掩码膨胀后的补集
    
    # 在语义一致+深度可行的区域内找最近点
    p' = argmin_{q ∈ C_p ∩ Sₜ} ||q - p||₂
    
    return p'
```

**参数**：
- D_max = 7m（最大深度阈值）
- τ = 0.5（相似度阈值）
- r = 0.2m（障碍物膨胀半径）

**效果**：
- 语义一致性：只在VLM特征相似区域内修正
- 几何安全性：确保修正后的点在障碍物安全距离外

#### 5.4 航向感知几何门控

**设计动机**：

当目标点靠近图像边界时，对应大的视角偏差，无人机应该**先转向**而不是**继续前进**。直接使用深度作为长程3D目标会导致：
- 规划器产生过多前进位移
- 偏航仍未对准目标航向
- 轨迹偏离预期路径

**门控算法**：

```python
def bearing_aware_gating(px', py', d', K):
    # 计算视角
    θ = arctan((px' - c_x) / f_x)
    
    # 门控前向距离（高斯衰减）
    d_f = d' * exp(-0.5 * (θ / (σ_θ * θ_max))²)
    
    # 反投影到3D
    u' = [px', py', 1]^T
    X_camera = d_f * K^(-1) * u'
    
    # 转换到里程计坐标系
    Gp = T @ X_camera
    
    return Gp
```

其中：
- θ：目标航向角
- θ_max = FOV_x / 2：水平视场角的一半
- σ_θ：控制距离衰减速度

**效果**：
- 目标在图像中心 → 接近全距离（d_f ≈ d'）
- 目标在图像边界 → 显著缩短距离（d_f << d'）
- 引导无人机先转向对准目标，再前进

### 6. 局部规划器（Local Planner）

#### 6.1 Fast Planner集成

使用**Fast Planner**生成短时域局部路径：
- 维护局部ESDF地图用于避障
- 强制UAV动力学约束，确保轨迹可执行

#### 6.2 偏航正则化

在避障过程中，将偏航对准目标航向（而非切向航向）：

```python
# 计算目标航向
ψ_g = atan2(Gp,y - yₜ, Gp,x - xₜ)

# 添加偏航惩罚
J_ψ = λ_ψ * ||wrap(ψₜ - ψ_g)||²
```

其中 wrap(·) 将角度映射到 (-π, π]。

**效果**：
- 避障时保持朝向目标
- 提高观测连续性
- 减少不必要的偏航旋转

## 创新点与改进

### 相比现有工作的改进

#### 1. 与监督AVLN方法的对比

| 方法 | 监督信号 | 泛化能力 | 局限性 |
|------|---------|---------|--------|
| 监督AVLN | 轨迹标注、航点标签 | 受限于训练数据分布 | 在开放环境分布偏移时性能下降 |
| **OnFly** | **零样本** | **强泛化** | **无需任务特定监督** |

#### 2. 与零样本AVLN方法的对比

##### TypeFly
- **方法**：LLM解析指令 + 目标检测器提供感知线索 + 调用UAV技能
- **问题**：
  - 过度依赖检测器泛化能力
  - 检测器无法覆盖目标场景时失败
- **OnFly改进**：直接使用VLM进行视觉推理，不依赖特定检测器

##### PIVOT
- **方法**：迭代视觉提示，在图像上叠加候选动作，让VLM选择
- **问题**：
  - VLM的间接使用，无法充分利用指令理解和视觉推理能力
  - 成功率有限，碰撞频繁
- **OnFly改进**：决策智能体直接预测导航目标，最大化VLM效用

##### SPF (See, Point, Fly)
- **方法**：将动作预测建模为2D航点grounding，转换为UAV运动命令
- **问题**：
  - 缺乏显式进度监控，难以判断何时到达目标
  - 安全与效率矛盾：
    - 高频短时域决策 → 避免碰撞但走走停停，效率低
    - 长程目标预测 → 提高效率但缺乏安全机制，易碰撞
- **OnFly改进**：
  - 双智能体架构解耦控制与监控
  - 语义-几何验证确保目标安全
  - 滚动时域规划平衡安全与效率

#### 3. 核心创新点总结

| 创新点 | 解决的问题 | 技术方案 |
|--------|-----------|---------|
| **共享感知双智能体** | 单流VLM决策不稳定、频率需求耦合 | 高频决策+低频监控，共享ViT特征，独立KV-cache |
| **混合记忆** | 滑动窗口丢失全局上下文、KV-cache不稳定 | 初始帧+关键帧+最新帧，前缀稳定序列化 |
| **语义-几何验证** | VLM目标不保证几何可行性 | 语义相似度掩码+深度可行性检查+航向门控 |
| **滚动时域规划** | 安全与效率矛盾 | ESDF避障+偏航正则化+优化轨迹生成 |

## 实验结果

### 仿真实验

#### 实验设置

**环境**：
- 10个高保真UE 4.27场景（城市街区、工厂、别墅、复式建筑等）
- 150个任务：
  - 物体导航：40%（到达指定目标）
  - 精确导航：40%（语义定位，如泳池边缘）
  - 长程导航：20%（跨多个区域）

**数据生成**：
- 真实轨迹来自专业飞行员的第一人称视角
- 任务描述由Qwen3-VL-235B-A22B生成并人工验证

**指标**：
- **SR（成功率）**：无碰撞停在目标区域内
- **OSR（Oracle成功率）**：轨迹上至少一次到达目标区域
- **CR（碰撞率）**：任务中是否发生碰撞
- **FT（飞行时间）**：从起飞到任务完成的总时间

**UAV参数**：
- 相机：前向，FOV 90°×60°，720p
- 最大速度：0.6 m/s
- 最大加速度：0.6 m/s²
- 最大偏航率：0.4 rad/s
- SR阈值：5m
- 时间限制：70s

**基线**：
- TypeFly：LLM+检测器+技能调用
- PIVOT：迭代视觉提示
- SPF：2D航点grounding

#### 主实验结果

| 方法 | SR (%) ↑ | OSR (%) ↑ | CR (%) ↓ | FT (s) ↓ |
|------|---------|----------|---------|---------|
| TypeFly | 5.7 | 17.5 | 66.5 | 35.4 |
| PIVOT | 10.3 | 44.2 | 64.1 | 32.9 |
| SPF | 26.4 | 61.5 | 42.7 | 39.2 |
| **OnFly** | **67.8** | **78.1** | **2.7** | **27.1** |

**关键发现**：

1. **TypeFly性能最差**：
   - SR仅5.7%，CR高达66.5%
   - 原因：过度依赖检测器，漏检导致规划和执行快速恶化

2. **PIVOT中等性能**：
   - SR 10.3%，CR 64.1%
   - 原因：VLM的间接使用限制了指令理解和视觉推理能力

3. **SPF相对较好**：
   - SR 26.4%，但CR仍42.7%，FT最长（39.2s）
   - 原因：缺乏进度监控，终止延迟；缺乏安全机制，碰撞多

4. **OnFly显著最优**：
   - SR提升到**67.8**（相比SPF的26.4%，提升**157%**）
   - CR降低到**2.7%**（相比SPF的42.7%，降低**94%**）
   - FT缩短到**27.1s**（相比SPF的39.2s，缩短**31%**）

#### 规划器控制实验

为了排除碰撞导致的失败，评估SPF+避障规划器：

| 方法 | SR (%) | OSR (%) | CR (%) | FT (s) |

## 局限性

### 1. VLM能力限制

**问题**：
- 依赖VLM的视觉理解和推理能力
- 在VLM训练数据外的场景可能性能下降
- 小模型（如2B）表现出过早停止和不稳定行为

**影响**：
- 泛化能力受限于VLM预训练数据
- 模型选择需权衡性能与效率

### 2. 长程任务挑战

**问题**：
- 虽然混合记忆改善了长程监控，但超长距离任务仍可能累积误差
- 关键帧预算固定（S=4），可能不足以覆盖极长轨迹

**影响**：
- 跨多个区域的超长程导航可能需要更多关键帧
- 可能需要调整关键帧选择策略

### 3. 动态环境适应性

**问题**：
- 当前系统主要针对静态环境设计
- 动态障碍物和移动目标需要额外的感知和规划机制

**影响**：
- 在高度动态环境（如人群密集区域）可能需要额外安全机制
- 动态目标跟踪（如行人跟踪）可能需要更频繁的重新规划

### 4. 传感器依赖

**问题**：
- 依赖RGB-D相机和LiDAR
- 传感器故障或极端环境（如强光、雨雪）可能影响性能

**影响**：
- 需要可靠的传感器硬件
- 恶劣天气条件下可能需要降级策略

### 5. 计算资源需求

**问题**：
- 需要Jetson Orin NX级别的边缘计算能力
- 小型无人机可能无法搭载足够计算资源

**影响**：
- 限制了可部署的无人机平台类型
- 更小型的无人机可能需要云端推理（牺牲实时性）

### 6. 指令理解局限

**问题**：
- 任务管理器依赖VLM一次性分解指令
- 复杂、模糊或矛盾的指令可能导致错误分解

**影响**：
- 需要清晰明确的自然语言指令
- 指令设计影响系统性能

## 三个核心问题回答

### 问题1：这篇文章的核心算法流程是怎样的？

OnFly的核心算法流程分为5个主要阶段：

#### 阶段1：任务分解
```
输入：长程自然语言指令 L
处理：任务管理器使用VLM将L分解为子任务序列 {L₀, L₁, ..., Lₙ}
输出：当前子任务 Lᵢ
```

#### 阶段2：双智能体决策
```
输入：当前子任务 Lᵢ、观测 Oₜ = {rgbₜ, depthₜ, poseₜ}

[决策智能体]（高频）：
  1. 构建决策提示词 P_dec（包含历史点线索）
  2. VLM预测图像目标点：(px, py) = VLM(P_dec, Lᵢ, Oₜ)
  3. 封装候选目标：Gc = [px, py, depthₜ, fₜ, poseₜ]

[监控智能体]（低频，2秒/次）：
  1. 构建混合记忆：Mₜ = [o₁, Qₜ, oₜ]
     - o₁：初始帧
     - Qₜ：关键帧列表（通过去重、分段选择、前缀稳定序列化构建）
     - oₜ：最新帧
  2. 构建监控提示词 P_mon
  3. VLM输出状态：sₜ = VLM(P_mon, Lᵢ, Mₜ) ∈ {CONTINUE, STOP, LOST}
  
输出：候选目标 Gc、状态信号 sₜ
```

#### 阶段3：语义-几何验证
```
输入：候选目标 Gc = [px, py, depthₜ, fₜ, poseₜ]

步骤1：语义引导目标修正
  1. 计算深度：d = depthₜ[py, px]
  2. 如果 d ≤ D_max（7m）：
     a. 计算ROI内的相似度掩码：M_p = {q | sim(fₜ[q], fₜ[p]) ≥ 0.5}
     b. 连通区域分析：C_p = CC(M_p, seed=(px, py))
     c. 深度可行性检查：Sₜ = 障碍物掩码膨胀(0.2m)的补集
     d. 找最近可行点：p' = argmin_{q ∈ C_p ∩ Sₜ} ||q - p||₂
  3. 否则：p' = p

步骤2：航向感知几何门控
  1. 计算视角：θ = arctan((px' - c_x) / f_x)
  2. 门控前向距离：d_f = d' * exp(-0.5 * (θ / (σ_θ * θ_max))²)
  3. 反投影到3D：X_camera = d_f * K^(-1) * [px', py', 1]^T
  4. 转换到里程计坐标系：Gp = T @ X_camera

输出：验证后的3D导航目标 Gp
```

#### 阶段4：滚动时域规划
```
输入：3D导航目标 Gp

处理：
  1. Fast Planner生成短时域局部路径
  2. 维护局部ESDF地图用于避障
  3. 强制UAV动力学约束（v_max=0.6m/s, a_max=0.6m/s², ω_max=0.4rad/s）
  4. 偏航正则化：对准目标航向而非切向航向
     J_ψ = λ_ψ * ||wrap(ψₜ - ψ_g)||²

输出：无碰撞可执行轨迹
```

#### 阶段5：执行与监控
```
循环执行：
  1. 执行轨迹段
  2. 监控智能体检查状态 sₜ：
     - CONTINUE：继续当前子任务
     - STOP：完成当前子任务
       * 如果是最后子任务 → 终止episode
       * 否则 → 推进到 Lᵢ₊₁
     - LOST：恢复程序
       * 停止飞行
       * 重新定向到上一个正常点的航向
```

**关键设计特点**：
1. **双智能体解耦**：高频决策（实时性）+ 低频监控（可靠性）
2. **混合记忆**：保留全局上下文 + KV-cache稳定
3. **语义-几何验证**：指令一致性 + 几何安全性
4. **滚动时域规划**：平衡安全与效率

### 问题2：这篇文章相比其他工作有什么改进和创新？

OnFly相比现有零样本AVLN方法（TypeFly、PIVOT、SPF）的改进和创新体现在以下6个方面：

#### 创新1：共享感知双智能体架构

**现有方法的问题**：
- **单流VLM决策**：高频控制与低频监控耦合，互相干扰
- **频率需求错位**：低频分析阻塞高频目标生成
- **多目标冲突**：不同任务共享上下文但目标不同

**OnFly的改进**：
```
传统单流架构：
  VLM → [控制+监控] → 目标 + 状态（互相干扰）

OnFly双智能体架构：
          共享ViT
             ↓
         缓存特征
         ↙       ↘
  [决策智能体]   [监控智能体]
  高频(实时)     低频(2s/次)
  独立KV-cache   独立KV-cache
```

**优势**：
- ✅ 消除频率需求耦合
- ✅ 避免推理干扰
- ✅ 共享感知减少冗余计算
- ✅ 独立推理保证各自性能

**实验验证**：
- 去除双智能体：SR从67.8%降至48.6%，FT从27.1s增至44.1s

---

#### 创新2：混合关键帧-最近帧记忆

**现有方法的问题**：
- **滑动窗口记忆**：
  - 新观测覆盖旧观测 → 丢失全局轨迹上下文
  - 频繁覆盖早期槽 → KV-cache前缀不稳定
  - 缓存命中率低 → 推理延迟高
  - 长程线索丢失 → 监控不可靠

**OnFly的改进**：
```
传统滑动窗口：
  [oₜ₋₄, oₜ₋₃, oₜ₋₂, oₜ₋₁, oₜ]  ← 新观测覆盖旧观测

OnFly混合记忆：
  [o₁, kf₁, kf₂, kf₃, kf₄, oₜ]
   ↑   ↑___________↑    ↑
  初始帧  关键帧列表   最新帧
  (固定)  (前缀稳定)   (实时)
```

**关键帧维护机制**：
1. **去重**：基于ViT特征相似度去除重复候选
2. **分段选择**：将轨迹距离分段，每段选一个代表关键帧
3. **粘性规则**：如果前一个胜出者仍在同一段，优先重用
4. **前缀稳定序列化**：保留最长有效前缀，确保KV-cache稳定

**优势**：
- ✅ 保留全局轨迹上下文（初始帧+关键帧）
- ✅ KV-cache前缀稳定 → 高缓存重用率
- ✅ 推理延迟低（监控延迟从7.47s降至1.15s）
- ✅ 可靠的长程监控（不会丢失早期信息）

**实验验证**：
- 相比滑动窗口：SR从60.4%提升到67.8%
- 相比时间采样：SR从64.2%提升到67.8%
- 监控延迟额外加速：1.98s → 1.15s（6.50倍加速）

---

#### 创新3：语义-几何验证器

**现有方法的问题**：
- **SPF**：VLM直接输出目标点，不保证几何可行性
  - 可能落在障碍物上
  - 可能在不可通行区域
- **纯几何修正**：可能违反指令语义

**OnFly的改进**：
```
SPF流程：
  VLM预测目标 → 直接执行（可能碰撞）

OnFly验证流程：
  VLM预测目标
       ↓
  [语义引导修正]
  只在VLM特征相似区域内修正
       ↓
  [航向感知门控]
  目标在边界时缩短距离
       ↓
  验证后的安全目标
```

**两步验证**：

1. **语义引导目标修正**：
   - 计算相似度掩码：只在语义一致区域内搜索
   - 连通区域分析：保留包含原始目标的区域
   - 深度可行性：避开障碍物（膨胀0.2m）

2. **航向感知几何门控**：
   - 目标靠近图像边界 → 大视角偏差
   - 高斯衰减前向距离：d_f = d' * exp(-0.5*(θ/θ_max)²)
   - 引导无人机先转向，再前进

**优势**：
- ✅ 语义一致性：修正不违反指令意图
- ✅ 几何安全性：避开障碍物
- ✅ 航向对准：避免轨迹偏离

**实验验证**：
- 去除验证器：SR从67.8%降至63.7%，OSR从78.1%降至72.6%

#### 创新6：自动进度监控与终止

**现有方法的问题**：
- **缺乏显式监控**：
  - 无法判断何时到达目标
  - 无法检测偏离轨迹
  - 无法自动终止或恢复

**OnFly的改进**：
```
现有方法：
  持续输出动作 → 无法判断完成 → 延迟终止或无限执行

OnFly：
  监控智能体 → 输出状态信号
    - CONTINUE：继续执行
    - STOP：任务完成
    - LOST：偏离轨迹 → 恢复程序
```

**状态管理**：
- **CONTINUE**：正常执行当前子任务
- **STOP**：
  - 完成当前子任务
  - 如果是最后子任务 → 终止episode
  - 否则 → 推进到下一个子任务
- **LOST**：
  - 无人机停止
  - 重新定向到上一个正常点的航向
  - 尝试恢复

**优势**：
- ✅ 及时终止（避免冗余动作）
- ✅ 检测失败（避免无限执行）
- ✅ 自动恢复（提高鲁棒性）

**实验验证**：
- SPF+规划器：OSR=80.3%（能到达目标），但SR=43.2%（无法及时停止）
- OnFly：OSR=78.1%，SR=67.8%（有效监控和终止）

## 机载运行效率

### 推理延迟分析

**测试平台**：Jetson Orin NX (16GB)

**VLM配置**：
- 模型：Qwen3-VL-4B-AWQ
- ViT：FP16（保持视觉精度）
- LLM：AWQ量化（提升效率）

#### 延迟对比

| 策略 | 决策延迟(s) | 加速比 | 监控延迟(s) | 加速比 |
|------|------------|--------|------------|--------|
| **Baseline**<br>（标准Transformers实现） | 3.83 | 1.00× | 7.47 | 1.00× |
| **Edge-Deploy**<br>（+TensorRT+AWQ+特征缓存+KV-cache重用+CUDA Graphs） | 0.81 | **4.73×** | 1.98 | **3.77×** |
| **w/ Hybrid Memory**<br>（Edge-Deploy + 混合记忆） | 0.81 | **4.73×** | 1.15 | **6.50×** |

#### 关键发现

1. **决策智能体延迟**：
   - Baseline：3.83秒
   - Edge-Deploy：0.81秒（**4.73倍加速**）
   - 满足实时控制需求（<1秒）

2. **监控智能体延迟**：
   - Baseline：7.47秒
   - Edge-Deploy：1.98秒（3.77倍加速）
   - +混合记忆：1.15秒（**6.50倍加速**）
   - 满足及时进度监控需求

3. **混合记忆的额外加速**：
   - 监控延迟从1.98s降至1.15s（**42%**提升）
   - 原因：KV-cache前缀稳定 → 更高的缓存命中率

| 优化技术 | 实现方式 | 效果 |

| **KV-cache重用** | 重用LLM的KV-cache | 减少冗余注意力计算 |
| **CUDA Graphs** | 使用CUDA Graphs优化解码 | 减少内核启动开销 |
