## 核心信息

### 摘要

多模态大语言模型(MLLMs)越来越多地用于执行视觉工作流,例如导航GUI,其中下一步取决于经过验证的视觉组合条件(例如,"如果出现权限对话框且界面颜色为绿色,则点击允许"),并且该过程可能分支或提前终止。然而,这一能力仍然评估不足:现有的基准测试集中于浅层组合或独立约束,而不是深度链接的组合条件。

在本文中,我们介绍了MM-CondChain,一个用于视觉基础深度组合推理的基准测试。每个基准实例都组织为一个多层推理链,其中每一层都包含一个基于视觉证据的非平凡组合条件,并由多个对象、属性或关系构建。为了正确回答,MLLM必须详细感知图像,在每个步骤推理多个视觉元素,并遵循结果执行路径到最终结果。

为了可扩展地构建此类工作流风格数据,我们提出了一个代理合成管道:规划器逐层生成组合条件,而可验证程序中间表示(VPIR)确保每一层的条件在机械上是可验证的。然后,合成器将这些验证过的层组装成完整的指令。使用此管道,我们在三个视觉领域构建基准:自然图像、数据图表和GUI轨迹。对一系列MLLM的实验表明,即使是最强的模型也仅获得53.33的Path F1,在硬负样本上急剧下降,并且随着深度或谓词复杂性的增长而进一步下降,证实了深度组合推理仍然是一个基本挑战。

### 关键贡献

1. **首个深度组合推理基准**: 引入MM-CondChain,第一个用于视觉基础深度组合推理的基准测试,具有多层控制流和链式硬负样本。

2. **VPIR代理合成管道**: 提出基于VPIR的代理合成管道,将逻辑构建与语言渲染解耦,实现可扩展的基准构建和机械可验证性。

3. **跨领域评估**: 在三个视觉领域实例化框架并评估十个MLLM,表明即使是最先进的模型也难以对组合视觉条件进行细粒度验证,特别是在硬负样本实例和更大的深度或谓词复杂性下。

## 核心技术发现

### 1. 视觉基础深度组合推理的挑战

#### 问题定义

当前MLLMs被期望超越简单的视觉问答,处理复杂的视觉工作流,其中正确的操作取决于一系列视觉检查:
- 如果出现对话框,验证它是否请求位置访问
- 如果是且应用程序受信任,点击允许
- 否则执行其他操作

这些任务需要**视觉基础深度组合推理**:
- 在每个步骤,模型必须验证多因素视觉条件
- 然后确定工作流是否继续或提前终止

#### 现有基准的不足

**第一个不足:组合深度**

现有视觉推理基准通常评估单层组合(例如,"对象是否为红色且大型?"):
- SugarCrepe、Winoground、ARO、MMComposition
- 指令遵循基准(IFEval、FollowBench、MIA-Bench、ComplexBench、MM-IFEval)专注于独立约束
- 两者都不需要模型跨层执行深度组合推理

**第二个不足:硬负样本的难度**

一些先前的基准包含用于组合理解的对比对:
- SugarCrepe、Winoground、ARO、Zhao et al.
- 但这些通常限于单层变化,例如替换一个属性或关系

### 2. VPIR (Verifiable Programmatic Intermediate Representation)

#### 核心思想

直接提示MLLM代理生成长、多层组合推理链通常导致逻辑不一致和不可验证的声明。为了解决这个问题,本文提出**可验证程序中间表示(VPIR)**:
- 首先构造可执行的、类Python的谓词
- 机械验证其对结构化视觉事实的真假
- 然后将验证过的逻辑翻译成自然语言

这使基准构建过程可靠、可控,并基于可验证的视觉证据。

#### VPIR的形式化定义

在层t,VPIR定义为一对可执行谓词程序:
- 真逻辑: p_t
- 反事实假逻辑: p̃_t

在沙箱执行环境Env(F_t)中验证这些谓词:
- 该环境仅暴露白名单内置操作符B(例如,len、set、all、any)
- 将每个事实键k∈V_t绑定到其提取的值F_t[k]

VPIR谓词的语义由其确定性布尔输出定义:
```
⟦p⟧(F_t) ≜ Exec(p; Env(F_t)) ∈ {0,1}
```

这保证绝对可验证性:
```
⟦p_t⟧(F_t) = 1, ⟦p̃_t⟧(F_t) = 0
```

#### 小例子

考虑一辆红色汽车停在蓝色卡车的左侧:
- 在层t,规划器选择r_t = Transition
- 提取器产生S_t = "汽车",F_t = {color: "red", position: "left"}
- 管道生成p_t: color == "red" and position == "left"
- 最小扰动p̃_t: color == "blue" and ...
- 翻译器渲染c_t = "汽车是红色的且在左侧"
- 机械执行确认⟦p_t⟧ = 1, ⟦p̃_t⟧ = 0

### 3. 代理合成管道 (Agentic Synthesis Pipeline)

#### 整体架构

给定多模态输入(自然图像、图表或GUI轨迹),管道迭代构建多层推理链:

1. **规划器(Planner)**: 决定是扩展、终止还是回滚链
2. **验证器(Verifier)**: 执行质量控制
3. **合成器(Composer)**: 将每个验证链编译成配对的基准实例

#### 逐层VPIR合成

每个层t通过四阶段工作流合成可验证的层逻辑:

**步骤1:关系策略和主题选择**

选择关系策略r_t ∈ R:
- **深化(Deepening)**: 通过放大其部分或新属性维度继续推理同一主题
- **转换(Transition)**: 通过空间/语义关系移动到不同但相关的实体

给定输入样本x和执行顺序链历史H_{t-1},实例化r_t为主题过滤器并构建可行候选集:
```
Ω_t ≜ Ω(x, H_{t-1}, r_t)
```

**步骤2:结构化事实提取**

为了防止逻辑合成期间的幻觉,管道基于结构化、领域无关的事实表示进行生成:

联合选择基础主题S_t ∈ Ω_t并产生主题-事实对:
```
(S_t, F_t) = E(x, r_t, H_{t-1})
```

提取的事实F_t构成类型化键值映射{(k, v_k)}:
- 每个键k表示视觉属性维度(例如,颜色、空间关系、计数、GUI状态)
- v_k是类型化观察(例如,红色、左侧、50、列表布局)

**两个关键设计原则**:
1. **以对象为中心的基础**: 主题S_t必须在视觉输入中唯一可定位
2. **结构优先表示**: 将F_t表示为JSON字典,定义程序命名空间V_t = keys(F_t)

**步骤3:VPIR生成**

建立事实空间F_t和变量命名空间V_t后,合成VPIR:
- 真逻辑p_t
- 反事实假逻辑p̃_t

通过在沙箱环境Env(F_t)中评估来正式验证:
- 确定性布尔输出
- 绝对可验证性保证

鼓励:
1. 非平凡的谓词复杂性(例如,具有嵌套结构和多个事实键的多子句布尔组合)
2. p̃_t相对于p_t的最小反事实扰动

**步骤4:逻辑渲染**

VPIR对(p_t, p̃_t)通过程序验证后,基于LLM的翻译器将可执行逻辑渲染为自然语言:
- 真条件文本c_t
- 反事实条件文本c̃_t

关键点:**真值锚定在代码执行中;语言仅是用于评估的表面渲染**。

### 4. 专用验证器(Dedicated Verifier)

采用基于MLLM的专用验证器进行集中质量控制。

在层t,候选是束B_t = (S_t, F_t, p_t, p̃_t, c_t, c̃_t)。

验证器返回结构化判决v = {passed, reasons, fix_hint}。

验证分两个阶段进行:

#### 阶段I:事实和主题验证

在任何语言渲染之前验证基础材料(S_t, F_t):

1. **视觉基础**: S_t必须在输入x中唯一可定位
2. **非重复性**: 主题和提取的事实不得重复H_{t-1}中的那些
3. **关系合规性**: 选择必须满足所选策略r_t
4. **模式与一致性**: F_t必须符合领域模式,具有一致的跨属性值

#### 阶段II:语言实现验证

根据验证的VPIR谓词(p_t, p̃_t)验证渲染的自然语言条件(c_t, c̃_t):

1. **语义保真度**: 自然语言必须保留VPIR逻辑,无残留代码伪影
2. **无歧义引用**: 每个子句必须明确命名其主题,避免共指歧义
3. **反事实质量**: c̃_t必须忠实反映p̃_t,同时保持与c_t的最小扰动

#### 反馈驱动的重新生成

验证是阶段感知的:
- 阶段I失败触发(S_t, F_t)的重新生成
- 阶段II失败保留验证的(S_t, F_t, p_t, p̃_t),仅重新渲染(c_t, c̃_t)

### 5. 规划器:验证感知链控制

引入验证感知规划器来管理链级控制流。

规划器与验证器之间的动态交互构成了管道的代理核心:
- 规划器提出动作
- 验证器提供反馈
- 规划器相应调整

在每个层t,规划器输出决策(a_t, r_t) = π(H_{t-1}),其中:
- a_t是动作
- r_t ∈ R是关系策略

**动作空间**:
1. **EXTEND**: 在提议策略r_t下合成新层
2. **FINISH**: 终止链并进行组合
3. **ROLLBACK**: 丢弃最近的非种子层并从验证前缀恢复

#### 混合深度控制

规划器结合硬编码规则和MLLM驱动策略。给定目标深度区间[d_min, d_max]:

- 如果depth(H_{t-1}) < d_min: 仅允许EXTEND
- 如果depth(H_{t-1}) > d_max: 仅允许FINISH或ROLLBACK
- 否则: 所有动作均允许

#### 验证感知回溯

规划器与验证器紧密耦合。当当前前沿出现重复验证失败时(例如,持续的主题重复或不可满足的关系约束),规划器触发ROLLBACK:
- 修剪失败层
- 从最后一个验证前缀恢复合成

此反馈循环防止管道陷入不可恢复状态。

### 6. 合成:配对路径指令编译

规划器发出FINISH后,获得包含T层的验证控制流骨架,其中每层t提供基础主题S_t及其真/反事实条件(c_t, c̃_t)。

由于控制流可能在任何层终止,在每个可能的退出点附加问题:
- 终端层的最终问题q^fin
- 每个中间层的辅助问题q_t^aux

所有问题都是具有确定性答案的多项选择题。与依赖LLM作为法官进行开放评估的先前复杂指令基准不同,此设计实现完全可重现和客观评分。

#### 步骤1:主题去泄漏

主题描述可能无意中揭示其关联条件。例如:
- 如果条件测试"汽车是否为红色"
- 将主题描述为"红色汽车"将泄漏答案

为了防止这种情况,基于MLLM的重写器通过移除条件揭示属性将每个S_t重述为安全主题S̄_t:
- 在需要时替换为替代的视觉基础描述符(例如,空间位置)
- 核心约束是S̄_t必须保持唯一引用性

#### 步骤2:配对路径实例化

从每个骨架,编译两个几乎同构的评估实例:

**真路径(True-path)**:
- 所有条件{c_t}_{t=1}^T成立
- 控制流到达终端层
- 正确答案对应q^fin

**假路径(False-path)**:
- 均匀采样分歧层j ∈ {1, ..., T-1}
- 交换c_j ← c̃_j
- 由于⟦p̃_j⟧(F_j) = 0,流在层j提前终止
- 正确答案变为q_j^aux

最后,将每个(S̄_t, c_t)合并为流畅的自然语言if子句,产生最终的嵌套指令。

此配对编译创建硬负样本:两个路径共享相同结构和几乎相同措辞,仅在隐藏在多个真实条件中的单个微妙扰动条件上不同。因此,区分它们需要对每个条件进行细粒度推理,而不是表面模式匹配。

### 7. 领域特定实例化

VPIR合成管道在核心上是领域无关的;领域特定的适应限于输入预处理和事实提取。在三个视觉领域实例化框架:

#### 自然图像

- **输入**: 单个图像
- **预处理**: 无
- **事实焦点**: 视觉属性(例如,颜色、空间关系)
- MLLM直接从原始图像提取开放模式视觉属性

#### 图表

- **输入**: 图像 + 元数据
- **预处理**: CSV对齐 + LLM修复
- **事实焦点**: 数值统计
- ChartQA注释经常出现x/y长度不匹配和零占位符伪影
- 应用确定性CSV对齐修复长度不一致
- LLM基于值提取修复缺失条目

#### GUI轨迹

- **输入**: 图像序列 + 注释
- **预处理**: 完整性 + CoAT解析
- **事实焦点**: 时间动作
- 验证轨迹完整性(确保屏幕截图计数匹配注释长度)
- 将CoAT动作描述解析为每步结构化字段(动作类型、目标元素、位置等)
- 将多图像序列传递给引擎

**关键洞察**: 核心组件(VPIR谓词生成、两阶段验证和规划器回溯)完全保持领域无关。领域特定代码限于输入适配器、事实构建器和策略注册表。

## 实验结果

### 数据统计

从三个视觉领域使用公开可用数据集构建MM-CondChain:

**自然领域**:
- 398张图像,来自SAM(204)和GQA(194)

**图表领域**:
- 200张图表图像,来自ChartQA
- 跨越条形图、折线图和饼图
- 具有结构化数值注释

**GUI领域**:
- 377个交互轨迹(总共3,421个屏幕截图,平均每轨迹9.07帧)
- 来自AITZ,提供AITW上的细粒度推理注释

**总计**:
- 975个评估样本
- 每个包含配对的真路径和假路径实例

### 提取事实和VPIR变量统计

观察到明显的领域特定模式:
- **自然实例**: 主要依赖对象属性和空间关系
- **图表实例**: 集中于数值和结构统计
- **GUI实例**: 强调动作、状态和轨迹级元数据

VPIR变量分布不仅仅镜像完整的提取事实分布。相反,VPIR选择性地重用提取属性的子集来组合可执行谓词,表明基准难度由基于基础视觉事实的结构化组合推理驱动,而不仅仅是原始属性频率。

### 逻辑模式统计

VPIR表达式显示出实质性的结构多样性:
- 前20个模板仅覆盖所有表达式的50.07%
- 需要128个唯一模板才能达到80%覆盖率
- 表明基准包含广泛的组合逻辑结构,而不是少量重复形式

主导模板本身已经结构复杂:
- 单个VPIR模板可以涉及多个谓词、嵌套逻辑操作符、可执行程序形式及其对应的自然语言渲染
- 正确解决这些实例不仅需要相关对象、属性和关系的视觉基础,还需要对这些视觉因素如何共同确定条件是否成立的组合推理

### 评估模型

评估一系列MLLM,跨越开源和专有系列:

**开源模型**:
- Qwen3-VL系列
- Qwen3.5系列
- GLM-4.6V系列
- Kimi-K2.5
- InternVL3系列
- InternVL3.5-8B

**专有模型**:
- GPT-4o-1120
- GPT-5-0807
- Gemini-2.5-Flash
- Gemini-2.5-Pro
- Gemini-3-Flash
- Gemini-3-Pro
- Qwen3-VL-Flash
- Qwen3-VL-Plus

### 评估指标

为每个领域报告三个指标:

1. **真路径准确率**: 模型正确遵循所有条件并选择对应最终问题q^fin的答案的真路径实例百分比

2. **假路径准确率**: 模型正确识别提前终止点并选择辅助答案q_j^aux的假路径实例百分比

3. **Path F1**: 真路径和假路径准确率的调和平均数,测量两个路径的平衡性能

还报告Avg(F1),三个领域Path F1的算术平均值,作为总体分数。

### 主要结果

**总体表现**:
- 当前MLLM仍然在MM-CondChain上挣扎
- Gemini-3-Pro取得最佳整体结果,53.33平均Path F1
- GPT-5-0807紧随其后,50.34
- 即使是最强的模型也仅略高于50 F1
- 表明多层控制流下的视觉基础深度组合推理对当前MLLM仍然高度挑战

**真路径 vs. 假路径**:
许多模型在真路径上的表现明显优于假路径:
- GPT-4o-1120: 自然领域83.92 vs. 12.81
- Qwen3.5-4B: 自然领域88.92 vs. 15.37
- Qwen3.5-9B: 自然领域91.69 vs. 13.10

此差距表明,在复杂多层条件下,模型倾向于过度假设条件成立,从而偏向"继续"分支。这种偏差在实际视觉工作流中可能存在风险,因为未能检测到违反条件可能导致模型在应该停止、切换分支或拒绝操作时继续进行。

**模型比较**:
- 专有模型在整体性能上通常优于开源模型
- Gemini-3-Pro和GPT-5-0807分别排名第一和第二
- 开源模型在特定设置中仍然具有竞争力
- Qwen3.5-397B-A17B在GUI上取得最佳分数(F1=40.19),超越该领域的所有专有模型
- Thinking模型通常优于其Instruct对应物,表明显式推理导向模型更适合此复杂基准

**领域难度**:
观察到明显的领域依赖难度:
- GUI是最具挑战性的领域:最佳F1仅40.19
- 低于自然(55.91)和图表(66.04)的最佳结果
- 原因可能是GUI实例需要推理多帧轨迹、用户动作和界面状态转换
- 许多图表条件一旦相关值被基础化,就简化为确定性数值比较

### 设计消融

#### 链深度的影响

构造具有控制最大深度2、4和6层的消融实例:

**结果**:
- 所有模型随着链深度增加表现出一致的性能下降
- 从深度2到深度6,Path F1在所有测试模型中相对下降约29-33%
- Gemini-3-Flash遭受最大相对下降(-33.2%),尽管取得最高绝对分数
- 表明即使强模型也难以随着顺序验证步骤数量的增长保持准确性

**洞察**:
- 跟踪多层条件逻辑对当前MLLM构成基本挑战
- 随深度近乎线性的退化表明错误跨层复合,而不是限于特定条件
- 强调MM-CondChain可配置深度设计用于探测顺序视觉推理极限的价值

#### 谓词复杂性的影响

对比两个VPIR生成设置:
- **简单谓词**: 最多2个逻辑操作符,至少2个属性键,无嵌套要求
- **复杂谓词**: 至少4个逻辑操作符,4个属性键和2个嵌套组

**结果**:
- 增加谓词复杂性导致所有模型的实质性性能下降
- 相对退化范围从27.7%到36.0%
- GPT-4o-1120遭受最大相对下降(-36.0%)
- 表明基线性能较弱的模型不成比例地受组合复杂性影响

**洞察**:
- 当前MLLM不仅难以跨层顺序推理(如深度消融所示)
- 而且难以在单个谓词内进行组合推理
- 两个维度 - 链深度和谓词复杂性 - 共同定义MM-CondChain的难度景观
- 实现模型能力的细粒度诊断

#### 总结

上述消融揭示了MM-CondChain中的两个正交难度轴:
1. **垂直复杂性**(链深度)
2. **水平复杂性**(层内谓词组合)

增加任一维度都会导致所有测试模型的一致且实质性性能下降,证实顺序推理和组合推理仍然是当前MLLM的基本瓶颈。

关键的是,这两个轴在基于VPIR的合成管道中独立可控,实现细粒度难度校准。此设计使MM-CondChain不仅作为评估基准,而且作为诊断工具,用于精确定位模型在视觉基础条件推理中失败的位置和原因。

## 个人思考

### 最令人兴奋的发现

#### 1. VPIR:逻辑与语言的解耦

**发现**: VPIR将逻辑构建与语言渲染解耦,这是基准构建的重要创新。

**为什么令人兴奋**:
- **可验证性**: 通过代码执行验证逻辑,避免LLM生成不可验证的声明
- **可控性**: 可以精确控制基准的难度和结构
- **可扩展性**: 管道可以自动化生成大量高质量基准实例

**对Spatial AGI的启发**:
- 可以使用类似的程序化表示来形式化空间逻辑
- 空间条件的真值可以通过几何计算确定性验证
- 这为Spatial AGI系统的可靠性提供了基础

#### 2. 硬负样本的最小扰动设计

**发现**: 通过翻转单个谓词创建硬负样本,两个路径几乎相同但执行结果不同。

**为什么令人兴奋**:
- **细粒度测试**: 强制模型精确验证每个条件,无法通过表面模式匹配解决
- **安全性**: 在安全关键应用(如机器人、自动驾驶)中,小错误可能导致严重后果
- **诊断价值**: 可以精确定位模型在哪个条件上失败

**对Spatial AGI的启发**:
- 在空间任务中,小扰动(如"左侧" vs "右侧")可能导致完全不同的操作
- 硬负样本测试对于确保Spatial AGI系统的安全性至关重要

#### 3. 真/假路径性能差距

**发现**: 模型在真路径上表现显著优于假路径,倾向于假设条件成立。

**为什么令人兴奋**:
- **揭示偏差**: 暴露了当前MLLM的系统性偏差
- **安全风险**: 在实际应用中,未能检测违反条件可能导致危险操作
- **改进方向**: 明确指出了需要改进的能力

**对Spatial AGI的启发**:
- Spatial AGI系统需要特别关注条件验证
- 不能假设空间条件总是成立
- 需要显式的"安全检查"机制

#### 4. 跨领域的一致性挑战

**发现**: 在三个不同领域(自然图像、图表、GUI)观察到一致的挑战模式。

**为什么令人兴奋**:
- **普遍性**: 深度组合推理是跨领域的普遍挑战
- **方法有效性**: VPIR方法在不同领域都有效
- **泛化能力**: 表明解决方案可能也具有跨领域泛化性

**对Spatial AGI的启发**:
- Spatial AGI的挑战可能也具有跨领域性
- 在不同空间场景(室内、室外、虚拟)中可能需要类似的方法

### 潜在局限

#### 1. 依赖事实提取的准确性

**局限**: VPIR的验证依赖于提取事实F_t的准确性。

**分析**:
- 如果事实提取错误,即使VPIR逻辑正确,验证结果也会错误
- 对于复杂场景(如拥挤场景、遮挡),准确提取事实可能困难
- 领域特定的事实提取器可能无法泛化

**影响**:
- 可能限制基准在复杂场景中的应用
- 需要为不同领域开发高质量的事实提取器

#### 2. 谓词复杂性的上限

**局限**: 虽然VPIR支持复杂谓词,但实际生成中复杂性有限。

**分析**:
- 太复杂的谓词可能难以理解和验证
- 自然语言渲染可能无法准确表达复杂逻辑
- 模型在复杂谓词上性能急剧下降

**影响**:
- 基准可能无法充分测试极端复杂的组合推理
- 需要平衡谓词复杂性和可理解性

#### 3. 多模态输入的限制

**局限**: 当前主要处理静态图像,对动态场景支持有限。

**分析**:
- GUI轨迹涉及多帧,但处理方式可能不够充分
- 缺少对视频、3D场景等的支持
- 时间推理能力测试不足

**影响**:
- 可能无法充分评估Spatial AGI在动态场景中的能力
- 需要扩展到更多模态

#### 4. 评估指标的限制

**局限**: 使用多项选择题评估,可能无法完全反映实际应用能力。

**分析**:
- 实际应用需要生成动作,而不仅是选择答案
- 多项选择可能高估模型能力
- 缺少对推理过程的评估

**影响**:
- 基准结果可能无法完全预测实际应用性能
- 需要结合其他评估方法
